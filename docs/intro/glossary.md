# プロジェクト用語集

> **注記**: このドキュメントはテンプレートの例示として提供するものです。

## アーキテクチャ用語

### Onion Architecture

外側のレイヤーが内側のレイヤーに依存し、内側のレイヤーは外側を知らないという単方向依存を持つアーキテクチャパターン。
本テンプレートでは CLI層（外側）→ ビジネスロジック層（内側）の方向に依存が流れる。
Python では Port/Adapter の境界を `Protocol` で実装する。

### CLI層

コマンドライン処理、実行時コンテキストの組み立て、エラーハンドリング、終了コード決定を担う最外殻のレイヤー。
Typer を使用し、`src/example/cli.py` がエントリーポイントとなる。
ビジネスロジックを含まず、Orchestrator への委譲のみを行う。

### ビジネスロジック層

ドメイン固有の処理とその制御フローを担うレイヤー。CLI層の内側に位置し、外部システム（DB・FS・外部API）への直接依存を持たない。
Orchestrator、Processor、Reader、Writer などのコンポーネントで構成される。
外部システムへの依存は Protocol 経由で抽象化し、テスト時はモックに差し替えられる。

### 基盤（Shared Kernel）

CLI層・ビジネスロジック層の両方から参照される横断的共通部品。
ソースコード上は `foundation/` パッケージとして配置する。
ファイルシステム・エラー処理・ロギング・データモデル基盤などの、特定機能に属さず安定した抽象を提供する。

機能固有の概念が混入した場合は `<feature>/` パッケージに移動する。

## 開発プロセス用語

### llms.txt

LLM（大規模言語モデル）が作業を開始する前に読み込むドキュメントインデックスファイル。
プロジェクトルートに配置し、タスク種別ごとに「どのドキュメントを読むべきか」を案内する。
LLM はこのファイルを起点に関連ドキュメントを特定し、設計思想を把握してからコード変更を行う。

### 仕様ドキュメント

各モジュールの要件定義と基本設計が記述されたドキュメント。
`docs/specs/`ディレクトリ配下で管理され、常に実装と同期する。
LLM へのコンテキスト提供を主目的とし、セッションをまたいで設計思想に沿った実装を促す。

### リファレンス実装

テンプレートが提供する、動作するサンプル実装。`src/example/` 配下のすべてがリファレンス実装であり、`tests/` 配下もテストコードのリファレンス実装として位置付ける。

新規機能パッケージを追加する際の規範（カノニカルな構造）として機能し、Onion Architecture と Composition Root + Orchestrator パターンを実際のコードで示す。

AI への例示としても機能する。高品質なコードを注入することで AI が生成するコードの品質も高まるという効果を意図して、徹底的にリファクタリングされている。
